# EIP-712 Signing for Reya Trading API

Clients that submit orders, cancels, or mass-cancels through the REST API produce EIP-712 signatures off-chain. The backend validates each signature against the payload before forwarding the request to the matching engine and/or the on-chain settlement layer.

Three distinct EIP-712 type families exist:

1. **`Order` / `OrderDetails`** â€” the signed order payload. Verified both off-chain (API) and on-chain (orders gateway) using the exact same typehash.
2. **`OrderCancel` / `OrderCancelDetails`** â€” signed order cancels. Matching-engine-layer, not forwarded on-chain.
3. **`MassCancel` / `MassCancelDetails`** â€” signed mass cancels. Matching-engine-layer, not forwarded on-chain.

All three share the same EIP-712 domain.

---

## Domain

```
EIP712Domain = {
  name:               "Reya",
  version:            "1",
  verifyingContract:  <OrdersGateway proxy address>
}
```

Note: `chainId` is **not** part of the domain. Instead, each signed envelope carries an explicit `verifyingChainId` field inside the struct so signatures remain portable across forks where the domain separator would otherwise diverge.

---

## Order (on-chain-verified)

The `Order` envelope wraps the flat `OrderDetails` struct with a chain id and a signature deadline.

### Typehash strings (exact bytes)

```
OrderDetails(
  uint128 accountId,
  uint128 marketId,
  uint128 exchangeId,
  uint8   orderType,
  int256  quantity,
  uint256 limitPrice,
  uint256 triggerPrice,
  uint8   timeInForce,
  uint64  clientOrderId,
  bool    reduceOnly,
  uint256 expiresAfter,
  address signer,
  uint256 nonce
)

Order(
  uint256       verifyingChainId,
  uint256       deadline,
  OrderDetails  order
)
```

Concrete type strings passed to `keccak256` (canonical EIP-712 encoding â€” no whitespace, single line):

```

```

### Field semantics

| Field          | Type     | Notes                                                                                                       |
|----------------|----------|-------------------------------------------------------------------------------------------------------------|
| `accountId`    | uint128  | Reya margin account id.                                                                                     |
| `marketId`     | uint128  | Unified market id. Perp: raw core market id. Spot: `coreMarketId + 1e10`. See [Market ID namespace](#market-id-namespace). |
| `exchangeId`   | uint128  | Exchange identifier. Must be non-zero.                                                                      |
| `orderType`    | uint8    | 0 = LIMIT, 1 = STOP_LOSS, 2 = TAKE_PROFIT.                                                                  |
| `quantity`     | int256   | Signed size. Positive = buy/long, negative = sell/short. The REST API surfaces this as `isBuy` + unsigned `qty`; the signed EIP-712 value is reconstructed as `isBuy ? +qty : -qty`. |
| `limitPrice`   | uint256  | Worst acceptable execution price. Must be non-zero.                                                         |
| `triggerPrice` | uint256  | Must be zero for LIMIT; must be non-zero for STOP_LOSS/TAKE_PROFIT.                                         |
| `timeInForce`  | uint8    | 0 = GTC, 1 = IOC. Signed but not on-chain-enforced.                                                         |
| `clientOrderId`| uint64   | Client-provided correlation id. Signed but not on-chain-enforced.                                           |
| `reduceOnly`   | bool     | Reduce-only intent. Must be false for spot markets.                                                         |
| `expiresAfter` | uint256  | Unix seconds. Order lifetime â€” enforced on-chain at execution. `0` = good-until-cancelled.                  |
| `signer`       | address  | Must match the address recovered from the signature.                                                        |
| `nonce`        | uint256  | Per-signer nonce. The off-chain API enforces strictly-increasing uniqueness per signer.                     |

### Envelope fields

| Field               | Type    | Notes                                                                                  |
|---------------------|---------|----------------------------------------------------------------------------------------|
| `verifyingChainId`  | uint256 | Reya chain id (e.g., `1729`).                                                          |
| `deadline`          | uint256 | Unix seconds. Signature validity deadline â€” enforced by the API at entry.              |
| `order`             | OrderDetails | Inner struct described above.                                                     |

### `deadline` vs `expiresAfter`

The signed payload carries two distinct time values:

- **`deadline`** (envelope) guards the signature itself. The API rejects any request whose `deadline` is in the past at the moment the order is received. It is not consulted on-chain.
- **`expiresAfter`** (inner `OrderDetails`) guards the order's lifetime in the book. The on-chain orders gateway rejects fills attempted after `expiresAfter`. The API does not enforce this on entry; it is purely an execution-time constraint.

This separation lets short-lived signatures back long-lived resting orders. Convention: when `expiresAfter > 0`, clients should set `deadline <= expiresAfter` â€” a signature that expires before the order does would leave the order unreachable after the deadline passes (the client would need to re-sign to modify/cancel it, defeating the point of the longer expiry). This invariant is a recommendation, not a protocol rule, and is intentionally not encoded as a JSON-schema constraint.

## Market ID namespace

Perp and spot markets share a single `uint128 marketId` field but live in disjoint numeric ranges:

- **Perp**: `marketId = coreMarketId` (raw).
- **Spot**: `marketId = coreMarketId + 1e10` (offset).

REST endpoints such as `GET /v2/markets` return the offset-encoded `marketId` for spot markets so clients can use it directly in the signed `OrderDetails.marketId` field.

## OrderCancel (matching-engine-layer)

Cancels and mass-cancels use distinct EIP-712 type strings and compact `uint64` field widths:

```
OrderCancelDetails(uint64 accountId,uint64 marketId,uint64 orderId,uint64 clOrdId,uint64 nonce)
OrderCancel(uint64 verifyingChainId,uint64 deadline,OrderCancelDetails cancel)
```

```
MassCancelDetails(uint64 accountId,uint64 marketId,uint64 nonce)
MassCancel(uint64 verifyingChainId,uint64 deadline,MassCancelDetails massCancel)
```

These signatures are verified by the matching engine (via the API) and are never forwarded on-chain.

---

## Replay protection

- Every signed order carries a per-signer `nonce`. The off-chain API rejects nonces that are less than or equal to the highest nonce previously accepted for that signer.
- `deadline` caps signature validity. Expired signatures are rejected.
- On-chain, the orders gateway additionally tracks a per-`(signer, nonce)` cumulative consumed quantity so a single signature cannot be replayed beyond its signed `quantity`.

---

## Signature format

All three families return a standard `(v, r, s)` ECDSA signature plus the `deadline` used to construct the digest. The REST API accepts the signature as a single hex string formed by `r || s || v`.
