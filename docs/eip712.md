# EIP-712 signing reference

Normative reference for clients that sign Reya order-entry payloads
(`createOrder`, `modifyOrder`, `cancelOrder`, `cancelAll`, `cancelAllAfter`)
without using the Reya TypeScript SDK. Every signed request carries an
EIP-712 signature over one of the envelopes below; the off-chain API
reconstructs the same typed-data digest and recovers the signer.

## Domain

The EIP-712 domain is **intentionally minimal** — it omits `chainId`:

```json
{
  "name": "Reya",
  "version": "1",
  "verifyingContract": "<OrdersGatewayProxy address for the target chain>"
}
```

- `name` = `"Reya"`, `version` = `"1"` (constants).
- `verifyingContract` = the **OrdersGatewayProxy** address for the chain you
  are trading on (NOT the matching engine, NOT the PassivePerpProxy). Use the
  proxy address published for that network.
- **`chainId` is deliberately omitted from the domain.** Chain binding is
  carried *inside each struct* as the first member, `verifyingChainId`, so the
  same domain separator is reused across the contract's helpers while the
  signature still commits to a specific chain. A signer MUST set
  `verifyingChainId` to the target chain id; the API rejects a signature whose
  recovered signer does not match for the configured chain.

## Market-id encoding (unified namespace)

The `marketId` signed in `OrderDetails` / the cancel structs uses the
**unified** wire namespace, not the raw on-chain id:

- **Perp markets:** `marketId = on-chain core market id` (no offset).
- **Spot markets:** `marketId = on-chain spot market id + 1e10`
  (`SPOT_MARKET_ID_OFFSET = 10_000_000_000`).

Sign the same unified id the REST/WS payload carries.

## Envelopes

Each envelope is `Name(verifyingChainId, deadline, <details>)` wrapping an
inner `*Details` struct. `deadline` is the **signature-validity** deadline
(Unix seconds) — it is NOT the order lifetime (`expiresAfter`) nor the
cancel-all-after countdown (`timeoutMs`).

### Order (createOrder, modifyOrder)

```
Order(uint256 verifyingChainId, uint256 deadline, OrderDetails order)
OrderDetails(
  uint128 accountId,
  uint128 marketId,
  uint128 exchangeId,
  uint8   orderType,      // 0 = LIMIT, 1 = STOP_LOSS, 2 = TAKE_PROFIT
  int256  quantity,       // signed: + = buy/long, - = sell/short
  uint256 limitPrice,     // E18
  uint256 triggerPrice,   // E18; 0 for LIMIT, non-zero for STOP_LOSS/TAKE_PROFIT
  uint8   timeInForce,    // 0 = GTC, 1 = IOC, 2 = GTT
  uint64  clientOrderId,  // 0 = no client tag
  bool    reduceOnly,
  bool    postOnly,
  uint256 expiresAfter,   // order lifetime (Unix seconds); 0 for GTC, future ts for GTT
  address signer,
  uint256 nonce
)
```

`modifyOrder` signs the **same** `Order` / `OrderDetails` envelope as
`createOrder`, with the post-modify values substituted (full restate). The
REST surface splits the signed `int256 quantity` into `isBuy` + unsigned `qty`
(`quantity = isBuy ? +qty : -qty`) and uses string decimals; widen each to E18
with exact fixed-point parsing (`parseUnits(value, 18)`), never float math.

### OrderCancel (cancelOrder)

```
OrderCancel(uint64 verifyingChainId, uint64 deadline, OrderCancelDetails cancel)
OrderCancelDetails(uint64 accountId, uint64 marketId, uint64 orderId, uint64 clOrdId, uint64 nonce)
```

### MassCancel (cancelAll)

```
MassCancel(uint64 verifyingChainId, uint64 deadline, MassCancelDetails massCancel)
MassCancelDetails(uint64 accountId, uint64 marketId, uint64 nonce)
```

### CancelAllAfter (cancelAllAfter)

```
CancelAllAfter(uint64 verifyingChainId, uint64 deadline, CancelAllAfterDetails cancelAllAfter)
CancelAllAfterDetails(uint64 accountId, uint64 timeoutMs, uint64 nonce)
```

`timeoutMs` is the **relative** countdown (milliseconds) — the matching engine
stamps the absolute deadline on its own clock. It is signed directly; the
absolute trigger time is never part of the signature.

## Signing algorithm

Standard EIP-712 typed-data signing:

1. Build the domain separator from the domain object above
   (`name`, `version`, `verifyingContract`; no `chainId`).
2. Compute the struct hash of the chosen envelope. The type hash is
   `keccak256` of the canonical type string (primary type followed by any
   referenced struct types in alphabetical order, no whitespace) — derive it
   from the field lists above; do not hard-code a copied hash.
3. `digest = keccak256(0x1901 ‖ domainSeparator ‖ structHash)`.
4. Sign `digest` with the account's signing key; serialize as 65-byte
   `r ‖ s ‖ v`.

Each request needs a **fresh `nonce`** (monotonically increasing per signer);
replayed nonces are rejected with `INVALID_NONCE_ERROR`.
